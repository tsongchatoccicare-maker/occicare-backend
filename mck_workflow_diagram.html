System Architecture

โครงสร้างระบบในมุม Module-to-Module · ไม่ใช่ระดับโค้ด แต่เป็นมุมมองการไหลของงาน

section 02 — flow A (cont.)

Sales → Operation → Lab / X-ray / Report

รายละเอียดสิ่งที่เกิดขึ้นในแต่ละขั้น

2.2 Sub-steps Sales

  1. CRM — Add Customer ผู้ใช้ sales01 สร้างลูกค้าใหม่ ระบุ company_name, contact_person, phone, job_type (ตรวจสุขภาพ · OS XRAY · ตรวจซ้ำ · เก็บอาหารตัวอย่าง · อบรม First Aid · Consult · อื่นๆ) บันทึก: parttime_db__customers   ·   UI: Pages.customers
  2. Quotation สร้างใบเสนอราคาที่อ้างถึง customer · เก็บราคา + valid_until UI: Pages.quotation   ·   PDF export พร้อม
  3. Project (status=Draft) เปิด Project เมื่อ Quotation ได้รับการอนุมัติ UI: Pages.sales   ·   ระบบจะ auto-generate project_code (เช่น MCK-2025-001)
  4. Project status → Closed เปลี่ยนสถานะเป็น "Closed" = ปิดการขาย พร้อมส่งต่อ Operation หลังจากนี้ส่ง Notification ให้ op01
  5. Handover Checkbox sales tick กล่อง "ส่งต่อ" ในตาราง Project · ✅ icon + วันที่ปรากฏ ถ้า untick → ดึงกลับให้ Sales

2.3 Sub-steps Operation

  1. Create JO (Single หรือ Multi-day) สร้างใบแจ้งงาน 1 ใบต่อ 1 วัน · เลือก Project + จำนวนวัน 1-7 (หรือ Custom N) Multi-day → loop สร้าง N records · day_no=1..N · total_days=N
  2. Add Stations + staff_list แต่ละ JO มีหลาย Station (ลงทะเบียน · เจาะเลือด · X-ray · ฯลฯ) · แต่ละ station ระบุ staff_count และ staff_list[] (รายคน) 1 station = N คน · ระบุ profession · wage_per_day · staff_type
  3. Onsite — ปฏิบัติงาน ทีมออกตรวจ · เก็บ specimen · ตรวจร่างกาย · บันทึก vital signs UI: Pages.op_onsite   ·   project status → Onsite

2.4 Sub-steps Lab / X-ray / Report

  1. Lab รับ specimen → ตรวจวิเคราะห์ → ออกผล · TAT 5-7 วัน alert ถ้าเกิน TAT (category: TAT-Lab)
  2. X-ray ฟิล์มถึงรังสีแพทย์ → อ่านผล → ส่งผล · TAT 3-5 วัน alert (category: X-Ray)
  3. Report รวมผลทุกอย่างเป็น Report ฉบับสมบูรณ์ → ส่งให้ลูกค้า · SLA 30 วัน alert (category: SLA-Report)
⚠️
Alert & Notification แต่ละ alert จะส่งให้เฉพาะ role ที่เกี่ยวข้อง · categoryToModule map ใน Dashboard กรองให้ user เห็นเฉพาะแจ้งเตือนของตน (admin เห็นทั้งหมด)

2.5 Project Status Flow

Draft→ Quoted→ Closed→ Onsite→ Lab→ Report→ Billing→ Completed

3.2 Sub-steps OPD & Medical

  1. OPD รับผู้ป่วย จาก Report — ตรวจร่างกายโดยแพทย์ · บันทึก vital signs UI: Pages.opd   ·   role opd01
  2. ค่าวิกฤต (Critical Values) หากผลตรวจมีค่าผิดปกติ (น้ำตาลสูง · ไขมันสูง · BP สูง) → refer ต่อแพทย์เฉพาะทาง Notification category: ค่าวิกฤต
  3. Medical Record เก็บเวชระเบียน · ประวัติแพทย์ · ผลตรวจ UI: Pages.medical   ·   role medical01
  4. Follow-up ถ้าจำเป็น → schedule นัดผู้ป่วยกลับมา · ส่ง Notification ทาง Line/Email

3.3 Sub-steps Billing

  1. ออก Invoice คำนวณค่าบริการตาม headcount × อัตรา + งานเสริม UI: Pages.billing   ·   role billing01
  2. ติดตามการชำระ สถานะ Pending → Paid · ถ้าเกิน 30 วัน → Alert Billing category: Billing
  3. Project Cycle ปิด เมื่อ Invoice paid → status: Completed → ปิด workflow ของ Project นี้
🔁
Loop Pending → Paid เส้นประในแผนภาพแสดงว่า Invoice ที่ค้างจ่ายจะ alert ทุก 30 วัน จนกว่าจะ Paid · ไม่ block flow อื่น แค่เตือน

4.2 Sub-steps

  1. RegisterPT.html (Public Form) ผู้สมัครเปิดฟอร์มจาก browser · กรอกข้อมูลส่วนตัว · upload เอกสาร (3 ใบ: ใบประกอบ · บัตร ปชช · หน้าบัญชีธ.) เก็บใน parttime_db__staff   ·   ไม่ต้อง login
  2. Pending → admin review admin/sales เปิดหน้า Part-Time → เห็นรายการรอตรวจ · ดูเอกสาร · approve หรือ reject
  3. Approve เปลี่ยน status=approved + work_status=Active · บันทึก approved_by + วันที่ UI: Pages.parttime
  4. Available Pool รายชื่อทั้งหมดที่ approved → ใช้เป็น autocomplete ใน JO Station Op เลือกได้แบบ dropdown
  5. Assign to JO Station ใน JO ระบุชื่อ + วิชาชีพ + ประเภท (Part-time) + wage_per_day 1 station มีหลายคนได้ (staff_list[])
  6. Work onsite ทำงานในวันที่ JO ระบุ · 1 JO = 1 วัน · ถ้า project หลายวัน → คนเดียวกันอาจมีหลาย JO
  7. คำนวณค่าจ้าง รวมต่อ Project, รวมข้าม Project ทุกใบ → แสดงเป็น 💰 ฿X,XXX badge
  8. รายงานประวัติ PT หน้า "📊 รายงานประวัติ PT" แสดงแต่ละคนทำงานกี่ Project, ราคารวม, klick chip ดูรายละเอียดได้ UI: Pages.parttime_history

7.2 Multi-person Station Loop

ทุก Station ตอนนี้รองรับ หลายคน ผ่าน array staff_list[] · เวลาคำนวณค่าจ้าง Op Report จะ loop ผ่านทุกคน:

  1. JO มี N stations · แต่ละ station มี staff_count คน · staff_list[] เก็บข้อมูลแต่ละคน (profession · staff_name · staff_type · wage_per_day · phone · remark)
  2. Op Report _calcByTypes() วน for each station → for each person · filter ตาม selTypes (เช่น "Part-time") · sum wage_per_day per person
  3. PDF Export รวมเป็น rowspan ใน detail table · 1 station ที่มี 3 คน → 3 rows + Station name 1 ครั้ง (rowspan)

คำศัพท์ที่ใช้ในระบบ + API ที่อ้างถึงในแผนภาพ

8.1 คำศัพท์

คำศัพท์ความหมาย
MCKMobile Checkup — การออกตรวจสุขภาพเคลื่อนที่หน้างาน
Projectหน่วยงานขายระดับสูงสุด — 1 ลูกค้า 1 รอบงานตรวจ มี status ตามขั้น
JO (Job Order)ใบแจ้งงาน — 1 ใบ = 1 วันออกตรวจ · Project หลายวันมีหลาย JO
Stationจุดตรวจในวันออกตรวจ เช่น ลงทะเบียน · เจาะเลือด · เอกซเรย์
staff_list[]Array ของบุคลากรที่ประจำ Station นั้น · 1 station มีหลายคนได้
Part-Time (PT)บุคลากรชั่วคราว · สมัครผ่าน RegisterPT.html · admin อนุมัติ
work_statusสถานะการใช้งานของ PT: Active / Give a chance / Blacklist
day_noลำดับวันใน Project ที่มีหลายวัน (1, 2, 3, ...)
Token (Assessment)รหัส 12 chars unique ต่อ Project ใช้ใน URL ของ QR
TATTurn-around Time — เวลาในการดำเนินการให้เสร็จ (Lab 5-7 วัน)
SLAService Level Agreement — Report ต้องส่งภายใน 30 วัน
Handoverการส่งต่อ Project จาก Sales → Operation · ผ่าน checkbox
Radar Chartกราฟสรุปคะแนนหลายมิติในรูปทรงหลายเหลี่ยม
Weakest (Top 3)คำถามที่คะแนนต่ำสุด 3 อันดับ — ใช้พัฒนาบริการ
